# AgentFirewall 项目目标与路线图

## 最终目标

**构建一个统一的运行时防火墙核心，能够在不改变策略语义、审批行为或审计格式的前提下，保护多个不同运行时（adapter）的 AI 系统。**

核心愿景：
- **一个核心，多个适配器**：不是为每个框架单独构建安全产品，而是提供一个可复用的运行时防火墙核心
- **标准化执行表面**：将不同运行时的行为转换为标准化的事件类型（prompt、tool_call、command、file_access、http_request）
- **一致的策略引擎**：所有适配器共享相同的策略包、审批流程和审计模式
- **可扩展的防护范围**：从 LangGraph 开始，逐步扩展到 OpenAI Agents SDK、MCP 客户端/服务器，以及更多工具调用运行时

## 当前状态 (v1.2.0)

**已发布**：
- ✅ 2 个官方支持的运行时适配器：LangGraph、OpenAI Agents SDK
- ✅ 核心防护功能：
  - 提示注入检测（37 种模式）
  - 工具调用审查/阻止
  -  guarded shell 执行（28 种危险命令模式）
  -  guarded 文件读写（27 种敏感路径模式）
  -  guarded 出站 HTTP 请求
- ✅ 7 条内置规则，开箱即用
- ✅ 审批处理器（终端交互、静态规则、自定义回调）
- ✅ 审计追踪（控制台、内存、JSONL 文件）
- ✅ 评估套件（LangGraph 19 个 case，OpenAI Agents 9 个 case，generic wrappers 7 个 case）
- ✅ 适配器兼容性契约、release gate 和 runtime support manifest

## 接下来重点要做什么

### 1.1 版本 - 适配器契约强化（已完成）

**目标**：让核心防火墙具备可复用性，使新适配器能够无缝接入

**已完成**：
- ✅ 适配器能力矩阵文档化
- ✅ OpenAI Agents SDK 适配器计划文档化
- ✅ OpenAI Agents SDK 评估基础设施创建
- ✅ OpenAI Agents SDK 集成模块创建
- ✅ 预览支持文档化

**结果**：
- ✅ 为第二个官方适配器建立了明确的契约基础
- ✅ 让 conformance、capability matrix 和 release gate 成为代码中的 source of truth

### 1.2 版本 - 第二个官方适配器（已发布）

**目标**：发布 OpenAI Agents SDK 作为官方支持的适配器

**已完成**：
- ✅ OpenAI Agents SDK 适配器的全面实现
- ✅ 通过一致性测试套件验证
- ✅ 保持与 LangGraph 相同的策略和审计行为
- ✅ 提供本地可复现的演示、评估和 release gate 证据

**成功标志**：
- 新用户可以在两个适配器上看到相同的 `allow`/`block`/`review`/`log` 流程
- 无需学习第二套策略模型

### 1.3 版本 - MCP 预览支持与资源表面

**目标**：在不破坏现有两条官方 adapter 契约的前提下，把 MCP client/server 路径接入共享核心，并先以 preview 方式交付

**计划**：
- 支持通过相同决策模型的 MCP 客户端工具调用
- 支持受保护的 MCP 服务器端工具处理器和资源读取器
- 为 MCP 资源读取增加共享的 `resource_access` 事件表面，而不是假装成文件或 HTTP
- 将 MCP 保持为 preview runtime support，而不是在 `1.3` 里直接宣称官方支持
- 保持所有证据本地可复现、无网络、无 hosted 依赖

**非目标**：
- ❌ 完整 MCP 生态兼容
- ❌ sampling、notifications、subscriptions、roots、prompts
- ❌ 任何 hosted MCP 平台或集中式控制面
- ❌ 在 `1.3` 里把 MCP 提升为第三个官方 adapter

### 1.4 版本 - 通用包装器

**目标**：为尚无官方适配器的运行时提供低级包装器

**计划**：
- 直接工具调度包装器
-  guarded 子进程包装器
-  guarded 文件系统包装器
-  guarded HTTP 包装器
- 让 `log-only` 成为新集成的默认验证路径

## 扩展顺序的优先级标准

选择新适配器时考虑：
1. **与工具调用工作流的重叠度**：是否有大量工具调用场景
2. **在执行前阻止副作用的能力**：能否在危险操作发生前进行拦截
3. **现有事件模型的复用度**：能否使用已有的标准化事件
4. **本地可复现性**：能否在没有外部服务的情况下演示和测试
5. **清晰的 `log-only` 采用路径**：能否先观察再强制执行

## 不做的事情

- ❌ 在适配器证据存在之前声称通用兼容性
- ❌ 按框架分叉策略逻辑
- ❌ 让运行时特定的营销领先于支持契约
- ❌ 在适配器层稳定之前添加集中式控制平面
- ❌ 在 1.0 阶段构建审查员 UI 或集中式审批服务
- ❌ 在 1.0 阶段进行超出默认策略包的生产级误报调优

## 长期愿景

**部署模式演进**：
1. **当前**：进程内 SDK 模式（最强执行路径上下文）
2. **未来**：探索 sidecar、代理或集中式管理部署（在 SDK 模式稳定后）

**生态系统建设**：

- 新规则表面的策略包改进
- 适配器兼容性和运行时集成强化

## 关键设计原则

1. **标准化执行表面，而非框架特定 API**
   - 每个新适配器将运行时行为转换为共享事件类型
   - 仅在真正出现新表面时才添加新事件类型

2. **核心策略引擎保持运行时无关**
   - 标准化事件、决策、策略包、审批流程、审计模式
   - 执行表面强制器在工具调度、shell、文件和出站 HTTP 边界共享

3. **适配器 SPI 小而明确**
   - 将运行时特定钩子转换为共享事件和运行时上下文元数据
   - 防止适配器特定语义泄漏到策略代码中

4. **渐进式采用路径**
   - `log-only` 模式允许用户在强制执行前先观察
   - 降低采用门槛，积累真实工作流证据

## 成功指标

**短期（1.1-1.2）**：
- 2 个官方适配器通过一致性测试
- 相同的策略包在两个适配器上行为一致
- 评估套件覆盖两个适配器

**中期（1.3-1.4）**：
- MCP client/server 预览支持上线，并保持支持边界清晰
- 通用包装器被社区采用
- 出现社区贡献的攻击工作流和误报案例

**长期（2.0+）**：
- 多个官方适配器形成生态系统
- 集中式部署模式探索
- 成为 AI 代理运行时安全的标准层

---

**总结**：AgentFirewall 的终极目标是成为 AI 代理执行路径中的标准安全层，通过统一的策略核心和可扩展的适配器架构，保护各种工具调用 AI 系统免受提示注入和危险副作用的威胁。当前重点已经从“完成第二个官方适配器”转向“让两条官方 adapter 路径稳定可用，并在此基础上继续扩展到 MCP 和更多运行时”。
